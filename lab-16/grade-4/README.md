## 📐 Алгоритм перемножения

Матрицы A, B, C хранятся **сплошным блоком** `int*` размера N×N (доступ `M[i*N + j]`).

Все T потоков получают свой диапазон строк матрицы C:
- поток `t` обрабатывает строки `[t·⌊N/T⌋, (t+1)·⌊N/T⌋)`,
- последнему достаются ещё и остаточные строки (если `N % T ≠ 0`).

```c
for (i = start_row; i < end_row; i++)
    for (j = 0; j < N; j++) {
        sum = 0;
        for (k = 0; k < N; k++)
            sum += A[i*N + k] * B[k*N + j];
        C[i*N + j] = sum;
    }
```

A и B заполнены единицами → все элементы C должны быть равны N. Ex8 для `N ≥ 5` сам проверяет это и печатает `OK / FAIL`.

### Что видно на графике

- **Время растёт примерно как N³** (тройной вложенный цикл). По оси Y логарифмическая шкала — кривые превращаются почти в прямые с одинаковым наклоном.
- **Ускорение от числа потоков** ярко выражено до 4–8 потоков (`128→256→512` падает в 2–4 раза), дальше плато: больше потоков, чем физических ядер CPU, не ускоряет — наоборот, борьба за кеш и переключение контекста добавляют накладные расходы.
- На **малых N (≤ 128)** многопоточная версия может быть **медленнее** — создание pthread занимает сотни микросекунд, а сама операция считает быстрее.
- На **больших N (≥ 512)** ускорение от 8 потоков к 16 уже почти не заметно — попадаем в потолок числа аппаратных ядер / SMT.

## 🛠 Подводные камни синхронизации

В `ex7_sync.c` мьютекс + condition variable — стандартный паттерн «передача эстафеты»:

```c
pthread_mutex_lock(&lock);
while (turn != MY_TURN)
    pthread_cond_wait(&cv, &lock);   /* отпускает lock, спит, возвращает lock */
/* критическая секция */
turn = OTHER_TURN;
pthread_cond_signal(&cv);
pthread_mutex_unlock(&lock);
```

> **Почему `while`, а не `if`?** `pthread_cond_wait` может «случайно» проснуться (spurious wakeup). Проверка условия в цикле страхует от этого.

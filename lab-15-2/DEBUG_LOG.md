# DEBUG_LOG — ход рассуждений

Для каждой задачи: симптом → инструмент → причина → фикс.

---

## task1 — segmentation fault при `printf("%d\n", arr[i])`

| | |
|---|---|
| **Симптом** | `Segmentation fault (core dumped)` сразу при первой итерации printf-цикла |
| **Инструмент** | AddressSanitizer показал: `SUMMARY: AddressSanitizer: SEGV ... 0x000000000000 → null pointer dereference` |
| **Причина** | Параметр `int *arr` в `init()` — копия указателя. `malloc` записывал адрес в копию, оригинал `arr` в `main` остался `NULL` |
| **Фикс** | `init()` возвращает указатель; `main` сохраняет результат в `arr`. Эквивалентный вариант: `int **arr`, не выбрал ради простоты |

## task2 — `number.num` после `format()` стал не 1025

| | |
|---|---|
| **Симптом** | `printf("num: %d\n", number.num)` напечатало `1024` (или `0`) |
| **Инструмент** | ASan показал `stack-buffer-overflow` в `sprintf` |
| **Причина** | `sprintf(str, "%3d", 1025)` пишет `'1','0','2','5','\0'` — 5 байт в буфер размера 4. Пятый байт `\0` попадает в соседнее поле `num` и обнуляет его младший байт |
| **Фикс** | Увеличить `str` до 16 байт. Заменить `sprintf` на `snprintf(str, sizeof str, ...)` — он гарантированно не пишет за границу |

## task3 — `z = 11` вместо ожидаемых 36

| | |
|---|---|
| **Симптом** | `(5 + 1)² = 36`, программа печатает `11` |
| **Инструмент** | `cpp -E task3.c` — посмотреть результат препроцессора |
| **Причина** | `#define SQR(x) x * x` → `SQR(y + 1)` превращается в `y + 1 * y + 1 = 5 + 5 + 1 = 11`. Приоритет `*` выше, чем `+` |
| **Фикс** | `#define SQR(x) ((x) * (x))` — оборачиваем и аргумент, и весь макрос |

## task4 — матрица не транспонируется

| | |
|---|---|
| **Симптом** | После `transpose()` матрица осталась как была: `1 2 3 / 4 5 6 / 7 8 9` |
| **Инструмент** | Ручная трассировка: распечатать матрицу после каждой итерации |
| **Причина** | Цикл `for i, j ∈ [0, N)` свопает каждую пару `(i,j) ↔ (j,i)` дважды — один раз при `i=0, j=1`, второй раз при `i=1, j=0`. Два свопа подряд = нет операции |
| **Фикс** | Внутренний цикл с `j = i + 1` — обрабатываем только верхний треугольник |

## task5 — криптографические алгоритмы

Здесь несколько мелких багов в разных функциях. Найдены преимущественно ASan-ом плюс сверкой с ожидаемым выводом из шапки файла.

### 5.1 `xor_encrypt`

- **Цикл `i <= text_len`** — читает байт-терминатор и пишет за границу `output`. Фикс: `i < text_len` + `output[text_len] = '\0'`.
- **Дополнительно в test_xor_cipher**: для дешифрования нельзя использовать `strlen` на шифре, потому что XOR может дать байт `0x00` в середине, и `strlen` оборвётся. Передаём длину явно.

### 5.2 `caesar_cipher`

- **Условие `> 'a' && < 'z'`** — пропускает крайние буквы `a` и `z`. Должно быть `>= 'a' && <= 'z'`.
- **Отрицательный shift** (например, `-3` при дешифровании): `(text - 'a' + shift) % 26` в C для отрицательного делимого даёт отрицательный результат. Фикс: предварительно `s = ((shift % 26) + 26) % 26`.
- **Терминатор** не ставился.

> ℹ️ Расхождение с ожидаемым выводом из методички: оригинал писал `lazy` → `oazb` (т.е. `a` и `z` НЕ сдвигаются — это и есть баг). После исправления выходит `lazy` → `odcb`, что **математически верно** для шифра Цезаря (`l+3=o, a+3=d, z+3=c, y+3=b`). Ожидание методички — некорректно.

### 5.3 `vigenere_encrypt`

- Ключ `"LEMON"` в верхнем регистре, а `shift = keyword[i] - 'a'` — даёт отрицательное значение. Фикс: проверка регистра ключа и текста через `isupper`.
- `output[i] = text[i] + shift` — нет операции `mod 26`. Символ может выйти за пределы алфавита. Фикс: `((text[i] - base + shift) % 26) + base`.

### 5.4 `simple_rc4`

- Первый цикл `for (i = 0; i < 256 + 32; i++)` обращается к `S[256..287]` — out-of-bounds. ASan ловит сразу. Должно быть `i < 256`.
- Второй цикл `for (k = 0; k <= data_len; k++)` — off-by-one.

### 5.5 `simple_hash`

- `sizeof(str)` — это `sizeof(char*)` = 8 (на x64), а не длина строки. Цикл срабатывает только на первых 8 байтах, плюс читает `\0` из-за `<=`. Фикс: `strlen(str)` и `<`.

### 5.6 `generate_key`

- `for (i = 0; i <= length; i++)` пишет в `key[length]` — за границу буфера на вызове `generate_key(key, 10)` с `char key[10]`. Фикс: `i < length - 1` и `key[length - 1] = '\0'`.

## task_star — 4G/5G симулятор (под звёздочку)

10+ багов разных классов. Все были найдены под `-fsanitize=address,undefined`.

| # | Функция | Класс ошибки | Фикс |
|---|---|---|---|
| 1 | `register_device` | off-by-one в `connected_cells` | `i < 5` вместо `i <= 5` |
| 2 | `calculate_signal_strength` | `log10(0) = -inf`, бесконечная рекурсия при d=0 | clamp `distance_km = max(0.01)`, убрана рекурсия |
| 2b | `calculate_signal_strength` | `normalized` без инициализации | инициализируем `= 0` |
| 3 | `calculate_network_metrics` | off-by-one `device_id > device_count` | `>=`, плюс инициализация `packet_loss = 0` |
| 4 | `handover_device` | off-by-one `device_index > device_count`, `i <= tower_count` | оба `>=` / `<` |
| 5 | `aggregate_traffic` | потенциальное деление на ноль `total/device_count` | проверка `users > 0` и `float` для накопления |
| 6 | `get_cell_status` | return указателя на стек (`status[50]`) | `static char status[64]` |
| 7 | `measure_qos` | `1 << 31` — signed UB | `unsigned int mask = 1u << 31` |
| 8 | `balance_load` | цикл по всему массиву (50 башен), плюс `i++` внутри | цикл по `tower_count`, единичный инкремент |
| 9 | `add_network_slice` | `realloc(p, n+1)` без `sizeof(char*)` | `realloc(p, (n+1) * sizeof(char*))` |
| 10 | `main` | разыменование неинициализированного `device_ptr` | удалено |
| 11 | `main` | double free `buffer` | удалён второй `free` |
| 12 | `main` | `strcpy` 50-байтной строки в `char[10]` | `snprintf` + `char[64]` |

После всех исправлений программа отрабатывает чисто:

```bash
$ gcc -fsanitize=address,undefined -g src/task_star.c -o /tmp/star -lm
$ /tmp/star
# (никаких сообщений санитайзеров)
```

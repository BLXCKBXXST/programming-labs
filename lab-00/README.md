## 🔍 Что внутри: ключевые моменты

### Задание 1. Форматирование

```
|555|
|     555|
|555     |
```

- `|%d|` — без выравнивания, ширина поля = длина числа.
- `|%8d|` — ширина поля 8, выравнивание **по правому краю** (по умолчанию).
- `|%-8d|` — ширина поля 8, выравнивание **по левому краю** (`-`).

```
a + b = 0.30000000000000004
```

> **Почему не 0.3?** Числа `0.1` и `0.2` нельзя точно представить в двоичной форме (`double` = IEEE-754 binary64). Их сумма даёт ближайшее представимое число — оно отличается от математических `0.3` на 2⁻⁵⁴. Спецификатор `%.17f` показывает все 17 значащих десятичных знаков, на которых эта ошибка видна.

### Задание 2. Переполнение

| Операция | Тип | Результат | Почему |
|---|---|---|---|
| `INT_MAX + 1` | `int` | `-2147483648` | формальное UB, на x86 обычно wrap к `INT_MIN` (two's complement) |
| `0u - 1` | `unsigned int` | `4294967295` | у `unsigned` wrap-around **определён** стандартом |
| `CHAR_MIN - 1` | `char` | `127` | то же UB, на x86 wrap `-128 → 127` |

### Задание 3. `%c` vs `%d`

```
? == 63
r == 114
b == c
```

`char ch = 63;` — в ASCII код `63` это символ `?`. `printf("%c", ch)` печатает символ, `printf("%d", ch)` — его числовой код. Та же переменная, разная интерпретация формата.

`short k = 99;` — `99` это ASCII-код буквы `c`. Поэтому `%c` от `k` напечатает `c`, даже хотя в коде записано число.

# Оценка 4 — Разбивка на три файла

## Задача

Та же дробь (вариант 16), но разделена на **три файла**:

| Файл | Содержимое |
|---|---|
| [main.c](main.c) | только функция `main()` |
| [fraction.c](fraction.c) | реализация `make/mul/div/print_fraction` |
| [fraction.h](fraction.h) | определение `typedef struct Fraction` + прототипы |

`main.c` подключает только `fraction.h` — не знает деталей реализации.

## Идея разделения

```
              ┌─────────────┐
              │  main.c     │  ← только main(), вызывает API
              └──────┬──────┘
                     │ #include "fraction.h"
                     ▼
              ┌─────────────┐
              │ fraction.h  │  ← typedef + прототипы
              └──────┬──────┘
                     │
                     ▼
              ┌─────────────┐
              │ fraction.c  │  ← тела функций
              └─────────────┘
```

При компиляции линкер сшивает `main.o` и `fraction.o`.

## Сборка

```bash
gcc main.c fraction.c -o fraction
./fraction
# Введите дробь 1 (a b): 1 2
# Введите дробь 2 (a b): 3 4
# f1 = 1/2
# f2 = 3/4
# f1 * f2 = 3/8
# f1 / f2 = 4/6
```

## Зачем это нужно

В реальных проектах редко всё держат в одном `.c` — теряется обзор кода. Разделение на header + источник позволяет:

- **Переиспользовать** `fraction.h` в другом проекте.
- **Скомпилировать раз** — линковать много раз.
- **Видеть API** в одном маленьком `.h` без шумного тела функций.

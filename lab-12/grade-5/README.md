- Собрать ещё и **динамическую библиотеку** (`.so`)

## Почему функции теперь с указателем?

В оценке 3 и 4 размер массива был фиксирован (`#define N 10`).  
Теперь нам нужно работать с разными размерами (100, 10000, 100000).  
Для этого выделяем память через `malloc` и передаём указатель на неё.

```c
Sneaker *arr = malloc(n * sizeof(Sneaker));  // выделяем память
// ... работаем с arr ...
free(arr);  // освобождаем — иначе утечка памяти!
```

## Как работает замер времени

```c
struct timeval t1, t2;
gettimeofday(&t1, NULL);   // запомнили момент ДО
BubbleSortByPriceN(arr, n);
gettimeofday(&t2, NULL);   // запомнили момент ПОСЛЕ

// считаем разницу в миллисекундах
double ms = (t2.tv_sec - t1.tv_sec) * 1000.0
          + (t2.tv_usec - t1.tv_usec) / 1000.0;
```

## Ожидаемые результаты замера

| N       | Примерное время |
|---------|-----------------|
| 100     | < 1 мс          |
| 10 000  | ~200 мс         |
| 100 000 | ~20 000 мс      |

BubbleSort очень медленный на больших данных — это нормально, он O(n²).

## Сборка

### Просто запустить:
```bash
gcc src/main.c src/sneakers.c -I include -o sneakers
./sneakers
```

### Собрать динамическую библиотеку (.so):
```bash
# шаг 1: компилируем sneakers.c в объектный файл с флагом -fPIC
gcc -c -fPIC src/sneakers.c -I include -o sneakers.o

# шаг 2: создаём .so файл
gcc -shared sneakers.o -o libsneakers.so

# шаг 3: линкуем main.c с .so библиотекой
gcc src/main.c -I include -L . -lsneakers -o sneakers

# шаг 4: указываем где искать .so при запуске
export LD_LIBRARY_PATH=.
./sneakers
```

## Чем статическая библиотека отличается от динамической?

| | Статическая (.a) | Динамическая (.so) |
|---|---|---|
| Код вшивается в exe? | ✅ Да | ❌ Нет, подгружается при запуске |
| Нужен файл .so рядом? | ❌ Нет | ✅ Да |
| Размер exe | Больше | Меньше |
| Обновить без пересборки? | ❌ Нет | ✅ Да |
